# FAME Pool-State Index

`society-bots` indexes latest reserve state for the reviewed FAME swap route universe exported by `www`. The registry artifact lives at `src/fame-swap-pool-state/registry/base-v1-pools.json` and is generated by `www` from its route candidates; do not hand-curate pool metadata in this repo.

## Scope

- Quote-model pools: Uniswap V2, volatile Solidly/Equalizer, and volatile Aerodrome V2.
- Tracked-only pools: stable Solidly, native wrap, Slipstream, Uniswap V3, Uniswap V4, and unknown unsupported invariants.
- No factory discovery, stable math, concentrated-liquidity replay, notifier behavior, or public solver routing is owned here.

## Runtime

- `FamePoolState` CDK creates one DynamoDB table keyed by `pk` and `sk`, a scheduled indexer Lambda, and an authenticated read API Lambda.
- The indexer scans Base `Sync` logs for quote-model pools, reconciles every quote-model pool with `getReserves` at the safe block, writes monotonic latest rows when reserves drift or rows are missing, updates `observedThroughBlock`, and advances a cursor only after reconciliation succeeds.
- The API serves bounded batch reads to server-side `www` callers. API Gateway uses a Lambda authorizer for the same service token before invoking the API Lambda, and the API Lambda keeps its own token check as defense in depth. The token must be supplied through `Authorization: Bearer <token>`.
- The indexer has SQS-backed async failure destinations plus passive CloudWatch alarms for Lambda errors, Lambda throttles, missed invocations, and non-empty failure queue depth. These alarms have no notification action by default; they are an inspectable health surface for a free community service.

## Indexer Flow

Each scheduled indexer invocation is one conservative pass over the generated quote-model pool set:

1. Read the current Base head from the configured RPC client.
2. Subtract `FAME_POOL_STATE_CONFIRMATION_BLOCKS`, default `2`, to choose a safe block.
3. Read the per-chain cursor from DynamoDB. If there is no cursor, start at the safe block so a first deploy seeds current state without historical backfill.
4. Fetch Sync logs from `cursor.observedThroughBlock + 1` through the safe block, capped so the range never starts after the safe block.
5. Read `getReserves` for every quote-model pool at the safe block before writing any freshness advancement.
6. Apply Sync log writes in chain order using block number, transaction index, and log index as the reserve event version.
7. Batch-read the latest rows and reconcile every quote-model pool against the safe-block `getReserves` snapshots.
8. Mark every pool observed through the safe block.
9. Advance the chain cursor through the safe block.

The ordering is intentional. If any `getReserves` call, DynamoDB write, or reconciliation step fails, the invocation fails before pool freshness and the cursor advance to the new safe block. Sync/reconciliation writes made during the run use the prior cursor freshness until the final observation phase, so callers do not see partially reconciled rows as fresh.

The deployed indexer Lambda is reserved to one concurrent invocation. Conditional writes already protect data monotonicity, but single concurrency avoids overlapping scheduled retries duplicating RPC and DynamoDB work for the same cursor range.

## Reserve Cache Model

The reserve cache is a latest-state read model, not a historical event store.

### Latest Rows

Each quote-model pool has one latest row:

- `pk`: `pool:<chainId>:<lowercase poolAddress>`
- `sk`: `latest`
- `poolId`: registry id from the current `www` artifact
- `chainId`, `poolAddress`, `token0`, `token1`
- `reserve0`, `reserve1`, and `k`, stored as decimal strings
- `lastReserveChangeBlock`, `lastEventTransactionIndex`, `lastEventLogIndex`, and optional `lastEventTransactionHash`
- `observedThroughBlock`
- `source`: `sync-event` or `getReserves`
- `sourceRegistryId`
- `updatedAt`

`lastReserveChangeBlock` tracks when reserves last changed. `observedThroughBlock` tracks how far the indexer proved the pool state. A quiet pool can have an old reserve-change block and still be fresh if `observedThroughBlock` is recent.

### Cursor Rows

Each chain has one cursor row:

- `pk`: `cursor:<chainId>:quote-model-v1`
- `sk`: `cursor`
- `chainId`
- `observedThroughBlock`
- `sourceRegistryId`
- `updatedAt`

The cursor is advanced only after every quote-model pool has been reconciled and observed through the same safe block.

### Write Rules

Latest-row writes are monotonic by both freshness and reserve event version:

- A write is accepted when the row is new.
- A write is accepted when it does not lower `observedThroughBlock` and its reserve event version is newer than the stored version.
- A write is ignored when an older overlapping invocation tries to replay stale data.
- The final observation update only increases `observedThroughBlock`; it never rewinds freshness.
- The cursor write only increases the chain cursor.

Sync events preserve precise event ordering. Reconciliation writes use `source: "getReserves"` and a same-block max event version, so a safe-block `getReserves` repair wins after all Sync logs through that safe block have already been considered.

Reconciliation repairs more than reserve drift. A row is rewritten when reserves differ, when a row is missing, when the registry id for the same pool address changes, or when the stored `sourceRegistryId` is stale. This keeps a quiet renamed pool from staying `unknown` until its next swap.

### API Reads

The API never discovers pools from DynamoDB. It first resolves request keys against the generated registry artifact, then batch-reads latest rows only for quote-model pools.

Accepted request keys are exact:

- `{ "poolId": "..." }`
- `{ "chainId": 8453, "poolAddress": "0x..." }`

Mixed keys or extra fields are `400` request errors. Unsupported registry entries return `unsupported` without a DynamoDB read. Quote-model rows are matched back to registry entries by chain id and pool address, not by persisted `poolId`, so registry id refreshes for the same address do not make valid rows disappear from API responses.

If DynamoDB returns `UnprocessedKeys`, the API treats the read as incomplete and returns a transport failure instead of pretending those rows are missing. `www` should then fall back to live reads.

## Freshness

Freshness is based on `observedThroughBlock`, not `lastReserveChangeBlock`.

- `observedThroughBlock` means the indexer has scanned safely through that block.
- `lastReserveChangeBlock` means the last block where reserves changed.
- Quiet pools can be fresh even when `lastReserveChangeBlock` is old.
- The producer default is `FAME_POOL_STATE_DEFAULT_MAX_FRESHNESS_BLOCKS`, currently deployed as `120` unless overridden.
- Callers may request stricter freshness; they cannot loosen the producer default.

## Statuses

- `fresh`: quote-model row exists and `currentBlock - observedThroughBlock <= effectiveMaxFreshnessBlocks`.
- `stale`: quote-model row exists but is outside the freshness bound, including rows whose `observedThroughBlock` is greater than the caller's `currentBlock`. `www` should fall back to live reads.
- `unknown`: pool is absent from the registry or has no indexed row yet.
- `unsupported`: pool is reviewed but not eligible for v1 local reserve replay.

## Logs

Indexer logs use `event: "fame-pool-state-indexed"` with chain id, block range, event counts, seeded pool count, reconciled pool count, observed pool count, duration, and registry id.

API logs use `event: "fame-pool-state-api-batch"` with registry id, current block, effective freshness, batch size, and status counts.

API transport failures are classified separately from per-pool fallback statuses:

- `400`: malformed JSON, invalid request shape, or batch limit violations.
- `401`: missing or incorrect service token.
- `5xx`: internal or dependency failure, including incomplete DynamoDB batch reads.

`fresh`, `stale`, `unknown`, and `unsupported` stay inside successful batch responses. `www` should treat non-successful transport responses as non-authoritative helper output and fall back to live reads.

## Passive Health Signals

Before enabling `www` production helper env, inspect the deployed stack for:

- Indexer Lambda `Errors` alarm: OK or no recent alarm state.
- Indexer Lambda `Throttles` alarm: OK or no recent alarm state.
- Indexer Lambda `Invocations` alarm: OK after the schedule has had time to run.
- Indexer failure queue depth alarm: OK and queue depth `0`.
- Recent `fame-pool-state-indexed` success logs with advancing or non-regressing `observedThroughBlock`.

Do not add email, pager, or chat notification actions for the first release unless the project ownership model changes.
